# Provision an EC2 instance using AWS CLI

## Overview

In this project, I used the AWS CLI to provision a complete EC2 environment from scratch. The work covered the full identity setup (creating a custom IAM policy, a group, a dedicated user, and an access key), the networking layer (a custom VPC, a subnet, an Internet Gateway, a route table, and a security group), and the EC2 layer (a key pair, an AMI lookup, the EC2 instance itself), and finished by connecting to the running instance over SSH.

The IAM portion was deliberately built on the principle of least privilege: an initial administrative user (created from the AWS Management Console) was used only to bootstrap a dedicated `ec2-builder-user` with a custom EC2-management policy, and that restricted user performed all the subsequent provisioning steps.

## Technologies Used

- **Cloud Provider:** AWS (IAM, VPC, EC2)
- **CLI Tool:** AWS CLI
- **Access Method:** SSH with key-based authentication
- **Operating System (target instance):** Amazon Linux 2023

## Initial Admin User Setup (AWS Management Console)

This section was used to bootstrap an administrative user with `AdministratorAccess`. The admin user is the bootstrapping identity used only to create the restricted `ec2-builder-user` later in this project. If an admin user is already available, skip directly to the [AWS CLI Installation and Configuration](#aws-cli-installation-and-configuration) section.

The steps below were performed in the IAM dashboard of the AWS Management Console (searched for "IAM" in the search bar to access it).

### Create the Admin Group and Attach the `AdministratorAccess` Policy

- Clicked on "IAM user groups" in the left-hand menu, then on "Create group".
- Entered a name for the group (e.g., `Admins`).
- In the "Attach permissions policies" section, searched for `AdministratorAccess` and checked the box next to it.
- Clicked on "Create user group" to create the group.

### Create the Admin User and Add It to the Admin Group

- Clicked on "IAM users" in the left-hand menu, then on "Create user".
- Entered a username (e.g., `admin-user`).
- Since this user will also operate on the AWS Management Console, selected the following options:
  - Checked "Provide user access to the AWS Management Console - optional".
  - Selected a password option, either "Autogenerated password" (and saved the password displayed by AWS) or "Custom password" (and entered the desired password).
  - Checked "Users must create a new password at next sign-in - Recommended" so that the user is forced to change the password at the first login.
- Clicked on "Next" to proceed to the permissions step.
- In the "Set permissions" step, selected "Add user to group" and checked the box next to the previously created admin group, which transitively granted the `AdministratorAccess` policy to the user.
- Clicked on "Next" to proceed to the "Review and create" step, then on "Create user".
- Saved the console password (or downloaded the `.csv` file containing the credentials), as they are required to log in to the AWS Management Console as this user.

### Create an Access Key for the Admin User

- On the user details page, clicked on the "Security credentials" tab. If the redirect did not happen automatically, navigated to the "IAM users" list, clicked on the user name, then on the "Security credentials" tab.
- Scrolled down to the "Access keys" section and clicked on "Create access key".
- In the pop-up, selected "Command Line Interface (CLI)" as the access key type, confirmed the recommendation displayed below, and clicked on "Next".
- Clicked on "Create access key" to generate the credentials, then saved the Access Key ID and Secret Access Key in a secure location, since they are required to configure the AWS CLI in the next section.

## AWS CLI Installation and Configuration

- Installed the AWS CLI on the local machine by following the [official AWS documentation](https://docs.aws.amazon.com/cli/latest/userguide/getting-started-install.html), which provides installers for macOS, Linux, and Windows.
- Configured the AWS CLI with the admin user's access key:

```sh
aws configure
```

- Provided the Access Key ID, the Secret Access Key, the default region (e.g., `us-east-1`), and the default output format (e.g., `json`) when prompted.

Note: From this point forward, all the provisioning steps were performed through the AWS CLI. The reference documentation is available at [https://docs.aws.amazon.com/cli/latest/index.html](https://docs.aws.amazon.com/cli/latest/index.html).

## Create a Custom Policy for EC2 Management

A policy is a JSON document that defines a set of permissions for an IAM user, group, or role. To respect the principle of least privilege, a custom policy named `EC2ManagementPolicy` was created with only the actions required to provision and manage the resources of this project (VPCs, subnets, Internet Gateways, route tables, security groups, key pairs, EC2 instances, tags, and AMI lookups):

```sh
aws iam create-policy \
    --policy-name EC2ManagementPolicy \
    --policy-document '{
        "Version": "2012-10-17",
        "Statement": [
            {
                "Sid": "VPC",
                "Effect": "Allow",
                "Action": [
                    "ec2:CreateVpc",
                    "ec2:DeleteVpc",
                    "ec2:DescribeVpcs",
                    "ec2:ModifyVpcAttribute"
                ],
                "Resource": "*"
            },
            {
                "Sid": "Subnet",
                "Effect": "Allow",
                "Action": [
                    "ec2:CreateSubnet",
                    "ec2:DeleteSubnet",
                    "ec2:DescribeSubnets",
                    "ec2:ModifySubnetAttribute"
                ],
                "Resource": "*"
            },
            {
                "Sid": "InternetGateway",
                "Effect": "Allow",
                "Action": [
                    "ec2:CreateInternetGateway",
                    "ec2:DeleteInternetGateway",
                    "ec2:AttachInternetGateway",
                    "ec2:DetachInternetGateway",
                    "ec2:DescribeInternetGateways"
                ],
                "Resource": "*"
            },
            {
                "Sid": "RouteTable",
                "Effect": "Allow",
                "Action": [
                    "ec2:CreateRouteTable",
                    "ec2:DeleteRouteTable",
                    "ec2:CreateRoute",
                    "ec2:DeleteRoute",
                    "ec2:AssociateRouteTable",
                    "ec2:DisassociateRouteTable",
                    "ec2:DescribeRouteTables"
                ],
                "Resource": "*"
            },
            {
                "Sid": "SecurityGroup",
                "Effect": "Allow",
                "Action": [
                    "ec2:CreateSecurityGroup",
                    "ec2:DeleteSecurityGroup",
                    "ec2:AuthorizeSecurityGroupIngress",
                    "ec2:AuthorizeSecurityGroupEgress",
                    "ec2:RevokeSecurityGroupIngress",
                    "ec2:RevokeSecurityGroupEgress",
                    "ec2:DescribeSecurityGroups"
                ],
                "Resource": "*"
            },
            {
                "Sid": "KeyPair",
                "Effect": "Allow",
                "Action": [
                    "ec2:CreateKeyPair",
                    "ec2:ImportKeyPair",
                    "ec2:DeleteKeyPair",
                    "ec2:DescribeKeyPairs"
                ],
                "Resource": "*"
            },
            {
                "Sid": "Instance",
                "Effect": "Allow",
                "Action": [
                    "ec2:RunInstances",
                    "ec2:TerminateInstances",
                    "ec2:StartInstances",
                    "ec2:StopInstances",
                    "ec2:RebootInstances",
                    "ec2:DescribeInstances",
                    "ec2:DescribeInstanceStatus",
                    "ec2:DescribeInstanceTypes"
                ],
                "Resource": "*"
            },
            {
                "Sid": "Tagging",
                "Effect": "Allow",
                "Action": [
                    "ec2:CreateTags",
                    "ec2:DeleteTags",
                    "ec2:DescribeTags"
                ],
                "Resource": "*"
            },
            {
                "Sid": "Lookups",
                "Effect": "Allow",
                "Action": [
                    "ec2:DescribeImages"
                ],
                "Resource": "*"
            }
        ]
    }'
```

Each `Statement` (identified by its `Sid`) groups the actions required to manage one type of resource.

The `"Resource": "*"` value used in every statement grants the listed actions across all resources in the account. This setting is required for many EC2 actions (e.g., `CreateVpc`, `CreateSubnet`, `RunInstances`) that do not support resource-level permissions and therefore reject narrower ARNs. For the actions that do support resource-level permissions, `"Resource": "*"` could be restricted further (e.g., to a specific VPC or subnet ARN), but that level of granularity was kept out of scope for this learning project.

The ARN returned by this command is needed in the next step to attach the policy to the group.

## Create a Group and Attach the Custom Policy

A group is a collection of IAM users that share a set of permissions. Attaching a policy to a group means that every user added to that group inherits the permissions defined in the policy, which is a convenient way to manage permissions at scale:

```sh
aws iam create-group \
    --group-name EC2Builders

aws iam attach-group-policy \
    --group-name EC2Builders \
    --policy-arn <your-policy-arn>
```

Note: Replace `<your-policy-arn>` with the ARN returned by the `create-policy` command in the previous section.

## Create a User and Add It to the Group

An IAM user is an entity that represents a person or a service interacting with AWS. By creating a new user and adding it to the `EC2Builders` group, the user transitively inherits the permissions defined in the `EC2ManagementPolicy` policy:

```sh
aws iam create-user \
    --user-name ec2-builder-user

aws iam add-user-to-group \
    --user-name ec2-builder-user \
    --group-name EC2Builders
```

The user is created without access to the AWS Management Console by default, which is acceptable here because the user will only be used programmatically through the AWS CLI.

## Create an Access Key for the User

An access key is a pair of credentials (Access Key ID + Secret Access Key) that lets the AWS CLI authenticate against AWS as a given user. Created the access key for `ec2-builder-user`:

```sh
aws iam create-access-key \
    --user-name ec2-builder-user
```

Saved the Access Key ID and the Secret Access Key in a secure location, as they are required to configure the AWS CLI in the next step.

## Reconfigure the AWS CLI With the New User Access Key

Reconfigured the AWS CLI to use the newly created `ec2-builder-user` access key, so that the remainder of the provisioning is performed under the restricted user instead of the admin:

```sh
aws configure
```

Provided the new Access Key ID, the new Secret Access Key, the default region (e.g., `us-east-1`), and the default output format (e.g., `json`) when prompted.

## Create and Configure a VPC

A VPC (Virtual Private Cloud) is an isolated virtual network inside AWS, with its own private IP range (CIDR block). Every EC2 instance must belong to a network, and creating a custom VPC (instead of using the default one) gives full control over IP addressing, isolation, and topology.

This operation is covered by the `VPC` statement of the custom policy created earlier (`CreateVpc`, `DeleteVpc`, `DescribeVpcs`, `ModifyVpcAttribute`).

Created the VPC and enabled DNS support and DNS hostnames on it:

```sh
aws ec2 create-vpc \
    --cidr-block 10.0.0.0/16 \
    --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=MyVPC}]'

aws ec2 modify-vpc-attribute \
    --vpc-id <your-vpc-id> \
    --enable-dns-support

aws ec2 modify-vpc-attribute \
    --vpc-id <your-vpc-id> \
    --enable-dns-hostnames
```

Note: Replace `<your-vpc-id>` with the VPC ID returned by `create-vpc`.

## Create and Configure a Subnet

A subnet is a range of IP addresses inside a VPC, bound to a single Availability Zone (AZ). EC2 instances are launched into a specific subnet, which controls their IP addressing and routing. An AZ is a cluster of data centers within a region, and spreading resources across multiple AZs is the basis of high availability on AWS.

This operation is covered by the `Subnet` statement of the custom policy created earlier (`CreateSubnet`, `DeleteSubnet`, `DescribeSubnets`, `ModifySubnetAttribute`).

Created the subnet inside the VPC:

```sh
aws ec2 create-subnet \
    --vpc-id <your-vpc-id> \
    --cidr-block 10.0.1.0/24
```

By default, a subnet is created with auto-assign public IP disabled, so instances launched into it do not receive a public IPv4 address unless one is requested explicitly at launch (e.g., `--associate-public-ip-address`) or attached afterward as an Elastic IP. Enabled auto-assign public IP on the subnet:

```sh
aws ec2 modify-subnet-attribute \
    --subnet-id <your-subnet-id> \
    --map-public-ip-on-launch
```

Note 1: Enabling auto-assign public IP on the subnet only affects instances launched after this change. Existing instances in the subnet do not retroactively receive a public IP unless they are stopped and started again.

Note 2: Having a public IP is not enough by itself to allow inbound traffic from the internet. An Internet Gateway attached to the VPC, a route in the route table directing traffic to that gateway, and security group rules allowing the desired inbound traffic are all required.

## Create and Configure an Internet Gateway

An Internet Gateway (IGW) is a VPC component that allows traffic between the VPC and the public internet. Without an IGW, EC2 instances cannot reach the internet, which means inbound SSH connections from outside AWS would also be impossible.

This operation is covered by the `InternetGateway` statement of the custom policy created earlier (`CreateInternetGateway`, `DeleteInternetGateway`, `AttachInternetGateway`, `DetachInternetGateway`, `DescribeInternetGateways`).

Created the Internet Gateway and attached it to the VPC:

```sh
aws ec2 create-internet-gateway \
    --tag-specifications 'ResourceType=internet-gateway,Tags=[{Key=Name,Value=MyInternetGateway}]'

aws ec2 attach-internet-gateway \
    --internet-gateway-id <your-igw-id> \
    --vpc-id <your-vpc-id>
```

## Create and Configure a Route Table

A route table is a set of rules (routes) that determines where network traffic from a subnet or gateway is directed. Creating a route table and associating it with the subnet enables control over the flow of traffic to and from the EC2 instances. To allow internet access, a route directing traffic destined for the internet (`0.0.0.0/0`) to the Internet Gateway must be added.

This operation is covered by the `RouteTable` statement of the custom policy created earlier (`CreateRouteTable`, `DeleteRouteTable`, `CreateRoute`, `DeleteRoute`, `AssociateRouteTable`, `DisassociateRouteTable`, `DescribeRouteTables`).

Created the route table, added a default route to the Internet Gateway, and associated the route table with the subnet:

```sh
aws ec2 create-route-table \
    --vpc-id <your-vpc-id> \
    --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=MyPublicRouteTable}]'

aws ec2 create-route \
    --route-table-id <your-route-table-id> \
    --destination-cidr-block 0.0.0.0/0 \
    --gateway-id <your-igw-id>

aws ec2 associate-route-table \
    --route-table-id <your-route-table-id> \
    --subnet-id <your-subnet-id>
```

## Create and Configure a Security Group

A security group is a virtual firewall that controls inbound and outbound traffic for EC2 instances. Defining rules on a security group allows or denies specific types of traffic to and from the instances attached to that group. For SSH access, an inbound rule that allows TCP traffic on port 22 from a specific source IP is required.

This operation is covered by the `SecurityGroup` statement of the custom policy created earlier (`CreateSecurityGroup`, `DeleteSecurityGroup`, `AuthorizeSecurityGroupIngress`, `AuthorizeSecurityGroupEgress`, `RevokeSecurityGroupIngress`, `RevokeSecurityGroupEgress`, `DescribeSecurityGroups`).

Created the security group and authorized inbound SSH from the local machine IP:

```sh
aws ec2 create-security-group \
    --group-name sgrp-ec2-instance \
    --description "Security group for my new ec2 instance" \
    --vpc-id <your-vpc-id>

aws ec2 authorize-security-group-ingress \
    --group-id <your-security-group-id> \
    --protocol tcp \
    --port 22 \
    --cidr <your-ip-address>/32
```

Note: Replace `<your-ip-address>` with the public IP of the local machine that will initiate the SSH connection. The `/32` CIDR notation restricts the rule to that single IP.

## Create and Configure a Key Pair

A key pair is a set of cryptographic credentials that allows secure SSH connections to EC2 instances without using a password. AWS keeps the public key and provides the private key once at creation time.

This operation is covered by the `KeyPair` statement of the custom policy created earlier (`CreateKeyPair`, `ImportKeyPair`, `DeleteKeyPair`, `DescribeKeyPairs`).

Created the key pair, saved the private key to a `.pem` file, and restricted the file permissions to read-only for the owner (a requirement of SSH):

```sh
aws ec2 create-key-pair \
    --key-name my-key-pair \
    --query 'KeyMaterial' \
    --output text > my-key-pair.pem

chmod 400 my-key-pair.pem
```

## Get the Latest Amazon Linux 2023 AMI ID

An AMI (Amazon Machine Image) is a pre-configured template containing the operating system and software configuration used to launch an EC2 instance. Using an AMI saves the time of installing and configuring everything from scratch.

This operation is covered by the `Lookups` statement of the custom policy created earlier (`DescribeImages`).

Retrieved the ID of the latest Amazon Linux 2023 x86_64 AMI in the target region:

```sh
aws ec2 describe-images \
    --region <your-aws-region> \
    --owners amazon \
    --filters "Name=name,Values=al2023-ami-2023.*-x86_64" \
    --query 'reverse(sort_by(Images, &CreationDate))[:1] | [0].[ImageId, Name]' \
    --output json
```

The query sorts AMIs by creation date in descending order and returns the ID and name of the most recent one.

## Create the EC2 Instance

An EC2 instance is the actual virtual machine. Launching an instance requires specifying the AMI for the operating system, the instance type for the hardware specifications, the key pair for SSH access, the security group for network access control, and the subnet where the instance will live.

This operation is covered by the `Instance` and `Tagging` statements of the custom policy created earlier (`RunInstances`, `TerminateInstances`, `StartInstances`, `StopInstances`, `RebootInstances`, `DescribeInstances`, `DescribeInstanceStatus`, `DescribeInstanceTypes`, `CreateTags`, `DeleteTags`, `DescribeTags`).

Launched the EC2 instance:

```sh
aws ec2 run-instances \
    --image-id <your-ami-id> \
    --count 1 \
    --instance-type t3.micro \
    --key-name my-key-pair \
    --security-group-ids <your-security-group-id> \
    --subnet-id <your-subnet-id> \
    --associate-public-ip-address \
    --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=my-ec2-instance}]'
```

Note: `--associate-public-ip-address` is redundant when the subnet already has `map-public-ip-on-launch` enabled, but it is kept here for explicitness so that the command works regardless of the subnet's default behavior.

## Get the Public IP Address of the EC2 Instance

Retrieved the public IP address of the running instance by filtering on its `Name` tag and instance state:

```sh
aws ec2 describe-instances \
    --filters \
        "Name=tag:Name,Values=my-ec2-instance" \
        "Name=instance-state-name,Values=running" \
    --query 'Reservations[0].Instances[0].PublicIpAddress' \
    --output text
```

## Connect to the EC2 Instance via SSH

Connected to the instance via SSH using the private key created earlier and `ec2-user` (the default SSH user for Amazon Linux 2023):

```sh
ssh -i my-key-pair.pem ec2-user@<your-ec2-instance-public-ip>
```
